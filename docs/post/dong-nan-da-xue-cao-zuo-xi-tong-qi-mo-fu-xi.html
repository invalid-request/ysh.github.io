东南大学操作系统期末复习

第一章 操作系统绪论考点梳理

一、操作系统OS的定义（第一张PPT）📖

三层核心定义（考试名词解释高频）

  1. 硬件管控视角：OS管控、协调多个用户/应用程序对计算机硬件资源的使用，是资源分配器/资源管理者（resource allocator/manager）。
  2. 分层中介视角：OS是应用程序与计算机硬件之间的中间软件层，隔绝硬件底层细节，应用不用直接操作硬件。

💡考点记忆：一句话简答：操作系统是管理软硬件资源、介于应用与硬件之间的系统软件。

二、现代操作系统共性与三大基本特征（第二张PPT）📌

1. 两大发展共性（历史考点，选择/填空）

  • 多道程序（Multiprogramming）：内存同时存放多个程序，CPU在一个程序等待IO空闲时切换执行其他程序，大幅提升CPU利用率，是并发思想的起源。
  • 分时系统（Time sharing）：把CPU时间切分成极短时间片，轮流分给多个联机用户，每个用户独占感一台主机，诞生人机交互。

区分：多道侧重提高硬件利用率（批处理），分时侧重多用户同时交互使用。

2. OS三大核心特征（期末必考名词解释）

  1. 并发（Concurrence）：宏观多个进程同时运行，微观CPU分时交替执行，是现代OS最本质特征；并行是多核同一时刻同时执行，并发≠并行（易错辨析）。
  2. 共享（Sharing）：软硬件资源被多个并发进程共同使用，分互斥共享（打印机）、同时访问共享（磁盘文件）两类。
  3. 虚拟（Virtual）：物理资源抽象出多个逻辑资源，例：虚拟内存（磁盘当内存用）、时分复用CPU。

⚠️必背：并发是共享和虚拟的前提。

三、用户态&内核态（第三张PPT，选择题+简答题重难点）⚙️

1. 两种运行模式

运行模式 别名 运行主体 权限
用户态(User mode) 目态 普通用户应用程序 只能执行非特权指令，不能直接操作硬件
内核态(Monitor/Kernel/Privileged mode) 管态/特权态 操作系统内核 可执行全部特权指令，管理硬件、资源

2. 模式切换规则（必考）

  1. 用户态→内核态：唯一合法途径是系统调用（System Call）、中断、异常；用户程序无法直接切内核态，防止非法篡改硬件。
  2. 内核态→用户态：系统调用执行结束返回时，CPU自动切回用户态。

易错提醒：普通函数调用不会切换CPU状态，只有系统调用才发生状态切换。

3. 特权指令：仅内核态可执行（例：开关中断、内存地址修改、IO硬件操控）

四、程序(Program)与进程(Process)区别（第四张PPT，名词解释高频）🔍

  1. 程序（Program）：被动实体（passive entity）
    存放在磁盘上的静态二进制文件，是指令代码集合，没有运行、不占用系统资源，同一个程序可以生成多个进程。
    例：notepad.exe存放在C盘，只是程序。
  2. 进程（Process）：主动实体（active entity）
    程序被加载进内存后正在运行的实例，动态实体，占用CPU、内存资源；单线程进程自带程序计数器PC，记录下一条待执行指令地址。

简答模板：程序静态存盘，进程是程序的一次动态执行过程。

本章期末高频考点小结💯

  1. 名词解释：操作系统、并发、共享、虚拟、进程、系统调用、多道程序、分时系统
  2. 简答：用户态内核态区别及切换时机；程序和进程的区别；三大特征关系
  3. 选择：并发和并行辨析、特权指令运行状态、多道/分时系统特点

第二章 OS结构考点梳理

一、操作系统九大通用服务（第一张PPT·选择/简答考点）📌

OS为用户和应用提供的基础公共服务，考试多选高频：

  1. User interface(UI) 用户接口：人与OS交互入口，后续细分CLI/GUI
  2. Program execution 程序执行：加载程序入内存、创建进程、调度运行、正常/异常终止
  3. I/O operations I/O操作：屏蔽硬件细节，应用不用直接操控外设，由OS管理磁盘、键盘等IO
  4. File-system manipulation 文件管理：创建/删除/读写文件、目录管理，是最常用服务之一
  5. Communications 进程通信：本机进程IPC、跨主机网络数据传输（Socket）
  6. Error detection 错误检测：软硬件故障、程序越界/除零等异常捕获，报错与处理
  7. Resource allocation 资源分配：CPU、内存、外设、文件等软硬件资源分配管理
  8. Accounting 记账统计：记录各进程CPU/内存占用时长，用于计费、性能分析
  9. Protection and security 安全保护：权限管控，防止非法访问资源、数据泄密篡改

二、三层接口：UI、系统调用System Call、API（第二张PPT·必考辨析）💡

1. UI（面向普通终端用户）

  • CLI命令行接口：如Linux bash、Windows cmd，输入命令（ls/cd）操作系统
  • GUI图形界面：Windows桌面、macOS桌面，鼠标点击可视化操作

本质：CLI/GUI最终底层都会调用系统调用完成功能

2. 系统调用System Call（OS内核提供的编程接口，用户态→内核态唯一通道，名词解释必背）

  • 定义：操作系统内核向外暴露的底层服务接口，是应用请求OS资源的唯一入口；用户程序发起系统调用触发陷入，CPU从用户态切换内核态。
  • 举例：Linux open()、read()、fork()，Windows创建进程API底层都封装系统调用。

3. API应用程序接口（面向程序员的开发接口）

  • 定义：编程语言/系统库封装的函数集合，API内部封装了系统调用，程序员不用直接写原生系统调用。
  • 层级关系：应用代码 → API → 系统调用 → OS内核
    例：C标准库fopen()是API，内部封装Linux的open系统调用。

⚠️易错区分：系统调用是内核原生接口；API是上层封装，一个API可能对应多个系统调用。

三、三种经典操作系统内核组织结构（后三张PPT，简答+对比大题）⚙️

1. 分层结构 Layered Approach（左环形示意图）

  • 架构：硬件在Layer0，逐层向上分层，上层仅能调用紧邻下层接口，不能跨层访问；最顶层LayerN是用户接口。
  • ✅优点：模块化、易调试、故障隔离，下层改动不影响上层
  • ❌缺点：层级过多带来额外开销，跨层交互效率低；经典案例：THE操作系统
  • 图示：同心圆由内向外：硬件→各层内核→用户接口

2. 微内核结构 Microkernel Approach（右侧Mach+BSD示意图，macOS/鸿蒙采用）

  • 架构：只把最核心功能（进程调度、IPC、基础内存管理）放进内核；文件系统、设备驱动、网络等服务全部放在用户态进程，服务间通过消息通信。
  • ✅优点：内核小巧稳定，驱动崩溃不会导致系统宕机；扩展性好
  • ❌缺点：用户态服务间消息通信开销大，性能偏弱；案例：Mach（苹果底层内核）、QNX
  • 图示：底层Mach微内核 + 上层BSD用户态服务，应用运行在最上层

3. 模块化结构 Modules Approach（Solaris模块图，现代Linux主流架构）

  • 架构：一个核心主内核，其余功能（驱动、文件系统、调度器）做成可动态加载内核模块，运行时按需挂载/卸载，模块间直接调用。
  • ✅优点：兼顾效率与灵活，不用重新编译内核即可加载新驱动，性能优于分层/微内核
  • ❌缺点：模块在内核态运行，异常模块可能破坏整个内核；案例：Linux、Solaris
  • 图示：中心core Solaris内核，外围挂载驱动、文件系统、调度类等可插拔模块

三种结构对比速记

结构 内核大小 运行效率 稳定性
分层 中等 偏低 高
微内核 极小 偏低 最高
模块化 偏大 最高 中等

  2. 简答：系统调用和API的关系；三种内核架构优缺点对比
  3. 选择：OS九大服务归类、CLI/GUI/系统调用层级关系

第三章 进程Process期末考点全梳理

一、进程五状态转换模型（第一张图，必考选择+画图题）📌

5种状态含义

  1. New（新建）：进程刚创建，OS分配PCB、资源初始化，还未进入就绪队列。
  2. Ready（就绪）：进程资源配齐，等待CPU调度，排在就绪队列。
  3. Running（运行）：进程占用CPU正在执行指令（单CPU同一时刻只有1个进程运行）。
  4. Waiting/Block（阻塞/等待）：进程等待I/O、事件，主动放弃CPU，进入阻塞队列。
  5. Terminated（终止）：进程执行完毕/异常退出，OS回收资源，PCB等待销毁。

状态切换触发条件（核心考点）

  • new → ready：admitted（OS接纳进程入就绪队列）
  • ready → running：scheduler dispatch（短程调度选中进程，分配CPU）
  • running → ready：interrupt（时间片到/外部中断，让出CPU）
  • running → waiting：I/O or event wait（进程发起IO/等待某事件，主动阻塞）
  • waiting → ready：I/O or event completion（IO完成、等待事件到达，唤醒进程）
  • running → terminated：exit（进程正常结束或异常退出）

⚠️易错：就绪不能直接进阻塞，阻塞不能直接进运行。

二、PCB进程控制块（第二张，名词解释+多选）💡

PCB是进程唯一标识，OS靠PCB管理进程，进程上下文全部存在PCB中，是进程存在的唯一标志。
PCB核心存储字段：

  1. process state：进程当前状态
  2. process number(PID)：进程唯一标识符
  3. program counter(PC)：下一条要执行指令地址
  4. registers：CPU通用寄存器（上下文切换时保存现场）
  5. memory limits：进程内存边界信息
  6. list of open files：进程打开的文件列表

简答：PCB作用：记录进程全部管理信息，是OS感知、调度进程的依据。

三、三级调度（第3、4张PPT，高频简答辨析）⚙️

调度类型 别名 调度对象 功能 存在场景
长程调度 作业调度 外存后备作业 从磁盘选作业调入内存→新建→就绪，决定多道程序度 批处理OS；分时系统(Windows/Linux)无长程调度
中程调度 挂起调度Swap 内存就绪/阻塞进程 内存紧张时，把进程换出到磁盘（挂起）；空闲时换入内存，动态调节内存负载 多数现代OS，解决内存不足
短程调度 CPU调度 就绪队列进程 从就绪选进程上CPU运行，频率最高（毫秒级） 所有多道OS，每次时钟中断/阻塞都会触发

上下文切换Context Switch

  1. 触发：CPU从旧进程切新进程时发生
  2. 操作：保存旧进程寄存器/PC等现场到PCB → 从新进程PCB加载上下文到CPU
  3. 考点：切换耗时属于纯系统开销，不产生用户有效计算。

四、进程创建：fork() + exec()（第五张，Linux必考）📝

子进程地址空间两种形式

  1. fork()：子进程复制父进程
    子进程和父进程拥有完全相同程序、数据，初始资源拷贝；fork调用成功：父进程返回子PID，子进程返回0。
  2. exec()：子进程加载全新程序
    fork之后调用exec，丢弃原有复制的程序，载入新可执行文件，修改子进程地址空间。

经典组合：fork()+exec()：先复刻父进程生成子进程，再替换子进程运行新程序（shell执行命令原理）。

五、进程通信IPC两种方式（第六张，对比简答）🔍

1. 消息传递Message passing（左图）

  • 流程：进程A发消息→内核缓冲区→进程B从内核取消息，数据需要经过内核中转两次拷贝
  • 优点：实现简单、隔离性好；缺点：数据拷贝多、速度慢
  • 适用：数据量小的进程通信

2. 共享内存Shared memory（右图）

  • 流程：OS开辟一块物理内存映射到A、B两个进程虚拟地址，进程直接读写共享区，不经过内核拷贝
  • 优点：无多余拷贝、通信速度最快；缺点：需要配合信号量实现互斥访问，同步复杂
  • 适用：大批量数据传输（如音视频、数据库）

速记：消息走内核、共享直访内存。

  2. 简答：五状态转换+触发条件；三级调度区别；fork与exec作用；两种IPC优缺点对比
  3. 选择：就绪/阻塞/运行切换正误判断、各级调度存在环境

第四章 线程Threads期末考点梳理

一、单线程 vs 多线程进程（第一张结构图📌）

资源共享规则（必考辨析）

  1. 同一进程内所有线程共享资源：代码(code)、全局数据(data)、打开文件列表(files)，这些资源存放在进程PCB中。
  2. 每个线程私有资源：独立栈(stack)、寄存器(registers)、程序计数器PC（线程切换只需要保存私有上下文，开销远小于进程切换）。
  • 左：单线程进程：1个进程=1个线程，全部资源唯一占用；
  • 右：多线程进程：1个进程包含多个线程，共享进程公共资源，各自拥有栈和寄存器。

线程核心优势

  1. 切换开销远低于进程（不用切换页表、内存空间）；
  2. 同进程线程通信简单（直接读写共享全局变量，无需IPC）；
  3. 提升CPU利用率，阻塞一个线程不影响同进程其他线程运行。

二、用户级&内核级线程四种映射模型（后四张图，简答+对比大题💡）

前置概念：
用户线程：运行在用户空间，由用户态线程库管理，OS内核不可见；
内核线程：由OS内核管理调度，内核感知，阻塞会触发CPU调度。

1. 多对一模型 Many-to-one（用户N→内核1）

  • 映射：多个用户线程映射同一个内核线程
    ✅优点：线程管理全在用户态，切换无系统调用、开销小
    ❌致命缺点：一个用户线程阻塞(IO)，整个内核线程阻塞，同进程所有线程全部阻塞；无法利用多核CPU并行
  • 代表：早期Solaris、GreenThread（Java老式绿色线程）

2. 一对一模型 One-to-one（用户1→内核1）

  • 映射：1个用户线程绑定1个独立内核线程
    ✅优点：单个线程阻塞不影响其他线程；天然支持多核CPU并行
    ❌缺点：创建用户线程需要创建对应内核线程，创建开销大、线程数量受限（内核TCB资源有限）
  • 代表：Windows、Linux原生pthread

3. 多对多模型 Many-to-many（用户N→内核M，N>M）

  • 映射：多个用户线程映射到少量内核线程，内核线程池数量可控
    ✅优点：兼顾前两者优点：①用户线程创建开销小；②单个用户线程阻塞，调度器切换同内核线程上其他用户线程，不卡死整组；可灵活配置内核线程数适配多核
    ❌缺点：实现逻辑复杂，需要用户态+内核态两层调度
  • 代表：Solaris、IRIX系统

4. 两级模型 Two-level（多对多变种）

在多对多基础上，部分用户线程可以一对一绑定专属内核线程，剩余用户线程复用内核线程池，兼顾灵活调度与特殊任务独占内核。

四种模型对比速记表

模型 阻塞影响 多核利用 线程创建开销
多对一 一线程阻塞→全阻塞 不能多核 极小
一对一 单线程阻塞互不影响 完美多核 大
多对多 局部阻塞，同池可切换 支持多核 中等
两级 混合特性 灵活多核 中等

本章高频考点总结✅

  1. 名词解释：用户级线程、内核级线程、线程上下文切换
  2. 简答：进程与线程区别；四种线程映射模型优缺点对比
  3. 选择：根据场景判断对应线程模型、线程私有/共享资源区分

第五章 CPU调度期末考点梳理

一、调度两大模式（第一张PPT📌）

1. 非剥夺调度（Non-preemptive）

进程一旦占用CPU，运行结束或主动阻塞才让出CPU，运行中新来高优先级进程无法打断当前进程。
代表算法：FCFS、非抢占式SJF、非抢占优先级调度。

2. 可剥夺调度（Preemptive，抢占式）

出现更优进程（新短作业/更高优先级进程）、时间片耗尽时，系统强行夺走当前CPU，切换就绪队列里的新进程。
代表算法：抢占SJF、抢占优先级、RR时间片轮转。

两个关键性能指标（计算题必考）

  1. 周转时间(Turnaround time) = 进程完成时刻 − 进程到达时刻（从进入系统到结束全耗时）
  2. 等待时间(Waiting time) = 进程在就绪队列排队总耗时（不包含CPU运行、IO阻塞时间）

考试核心：平均周转时间、平均等待时间是算法优劣评判标准。

二、四大经典调度算法（第二张PPT，计算题高频💡）

1. FCFS 先来先服务

  • 规则：按到达就绪队列先后顺序依次占用CPU，非抢占
    ✅优点：实现最简单、公平；❌缺点：短作业饥饿（短作业排在长作业后需要长时间等待），对短作业不友好。

2. SJF 短作业优先

  • 规则：优先选预估CPU运行时间最短的进程；分非抢占SJF、抢占SJF（最短剩余时间SRTF）
    ✅优点：平均等待时间四大算法里最优；❌致命缺点：饥饿(Starvation)，源源不断短作业到来，长作业永远得不到CPU。

3. 优先级调度(Priority)

  • 规则：优先调度优先级数值最高进程；分抢占/非抢占；数值越小优先级越高（常规约定）
    ✅优点：紧急高优任务优先运行；❌缺点：低优先级进程长期得不到调度→饥饿；解决方案：老化(Aging)，随等待时间逐步提升低优先级进程优先级。

4. RR 时间片轮转调度（分时系统标配，Windows/Linux）

  • 规则：就绪进程按FCFS排队，CPU每次只分配固定时间片(time slice)；进程用完时间片立刻被剥夺CPU，放回就绪队尾。
    ✅优点：所有进程按时间片轮流使用CPU，交互友好，无饥饿；
    ⚠️关键：时间片大小影响性能：时间片>>平均作业时长≈FCFS；时间片过小→频繁上下文切换，系统开销飙升。

三、配套专业名词（第三张PPT⚙️）

  1. Gantt chart甘特图：调度计算题专用图表，横轴为时间，分段标注各进程占用CPU时段，用来统计完成时刻、算周转/等待时间。
  2. Starvation饥饿：进程长期就绪，但始终无法被调度上CPU（SJF、静态优先级典型问题，老化Aging解决）。
  3. Time slice(Time quantum)时间片：RR算法单次分配给进程的CPU固定时长。

本章考试重点总结✅

  1. 名词解释：抢占调度、周转时间、饥饿、时间片、SJF、RR
  2. 大题：给到达时间+服务时间，分别用FCFS/SJF/RR画甘特图、计算平均周转时间、平均等待时间（必考计算题）
  3. 选择：区分抢占/非抢占、各算法优缺点、饥饿发生场景

第六章 进程同步期末考点全梳理

一、临界区 & 临界区三大准则（第一张PPT📌）

1. 临界区定义选择题

题目答案：D（访问临界资源的一段程序）

临界资源：多进程互斥访问的共享资源（打印机、共享变量）；临界区：操作临界资源的代码片段。

临界区问题三大必要准则（简答必背）

  1. Mutual Exclusion 互斥：同一时刻最多一个进程进入临界区，别的进程不能同时进入。
  2. Progress 空闲让进（推进）：临界区空闲时，等待的就绪进程可立即入选进入，不能无限阻塞。
  3. Bounded Waiting 有限等待：进程申请进入临界区后，等待进入的时间存在上界，不会无限饿死。

二、三种临界区实现方案（软件→硬件→信号量，2~6页PPT💡）

1. Peterson算法（软件方案，仅支持双进程）

flag[j]=true;    // j进程标记自己想要进入临界区
turn=i;          // 把优先权限交给对方i进程
while(flag[i]==true && turn==i); // 对方想进+优先权在对方则循环等待
临界区代码;
flag[j]=false;   // 退出，释放标记

✅优势：同时满足互斥、推进、有限等待三条准则；仅适用于两个进程。

2. TestAndSet() 硬件原子指令（自旋锁实现，硬件方案）

boolean TestAndSet(boolean lock){
    boolean rv = lock;
    lock = TRUE;
    return rv;
}
// 进程进入临界区逻辑
while(TestAndSet(lock)); // 自旋忙等，拿到锁才跳出循环
临界区;
lock = false;
  • Busy waiting（忙等/自旋Spinlock）：进程拿不到锁时，CPU空循环轮询，不放弃CPU。
    ✅多核优势：锁持有时间短时，无需上下文切换，开销远低于阻塞；
    ❌缺点：单CPU忙等浪费CPU算力。

3. 信号量Semaphore（主流方案，分互斥/同步信号量）

（1）互斥用法（初值=1，mutex）

Semaphore mutex=1;
do{
    wait(mutex);  // S.value--，<0则进程阻塞入等待队列
    临界区;
    signal(mutex);// S.value++，≤0则唤醒队列里一个进程
}while(true);

wait、signal原语底层逻辑

wait(S):
    S.value--;
    if(S.value<0){  // 资源不足，进程阻塞
        进程加入S的等待队列;
        block();
    }
signal(S):
    S.value++;
    if(S.value<=0){ // 队列存在等待进程，唤醒一个
        移出队首进程P;
        wakeup(P);
    }

区分：

  • 整型信号量（忙等）：wait循环忙等；
  • 记录型信号量（阻塞型，课本默认）：wait不足时进程阻塞放弃CPU，无忙等。

三、死锁Deadlock & 饥饿Starvation（7、8页⚙️）

1. 死锁案例（S、Q初值都=1）

$P_0$：wait(S); wait(Q);
$P_1$：wait(Q); wait(S);

  • 极端场景：$P_0$拿到S、$P_1$拿到Q，后续双方互相等待对方持有的资源，永久阻塞→死锁。

死锁四大必要条件：互斥、占有并等待、非剥夺、环路等待。

2. 饥饿Starvation（无限阻塞infinite blocking）

进程长期停留在信号量等待队列，永远无法被唤醒进入临界区；
成因：信号量唤醒策略不合理（如永远优先唤醒新来进程），老进程一直得不到资源；
解决：FCF队列、进程老化Aging。

四、三大经典同步问题（最后一页，大题必考）

  1. 生产者-消费者（有限缓冲Bounded-Buffer）：同步+互斥混合，空缓冲区empty、满缓冲区full同步信号量，buffer互斥mutex。
  2. 读者写者Readers-Writers：分读者优先/写者优先，解决：读可并发、写必须独占资源。
  3. 哲学家进餐Dining-Philosophers：5哲学家5筷子，解决死锁（最多4人申请筷子、左右手顺序颠倒、拿一双再进餐）。

五、管程(Monitor) + 条件变量(Condition variable)考点梳理💡

（一）管程 Monitor 基础

1. 诞生背景

信号量（wait/signal）编程易错：wait/signal写颠倒、漏写会直接死锁/破坏互斥；管程封装同步逻辑，由编译器自动保证管程内同一时刻仅一个进程执行，天然实现互斥，不用程序员手动加互斥锁。

  • 管程封装内容：共享数据 + 操作数据的过程(函数) + 局部变量；
  • 核心规则：任意时刻最多一个进程在管程内部运行，其他进入管程的进程阻塞在管程入口等待队列。

2. 条件变量 Condition variable

管程解决同步依赖（如生产者空缓冲区、消费者空产品）的配套机制，一个条件变量绑定一条独立等待队列，两个原语：

  1. wait(c)：当前进程释放管程、离开管程，阻塞到c对应的等待队列；别的进程可进入管程。
  2. signal(c)：唤醒c队列上一个等待进程；若队列为空，signal无任何效果（无遗留信号）。

⚠️区分：条件变量≠信号量，条件变量没有数值，仅用于阻塞/唤醒等待特定条件的进程。

（二）管程解决经典同步（举例：生产者消费者）

  1. 设两个条件变量：not_full(缓冲区未满，生产者等)、not_empty(缓冲区非空，消费者等)
  2. 生产者：缓冲区满 → wait(not_full)；放产品 → signal(not_empty)
  3. 消费者：缓冲区空 → wait(not_empty)；取产品 → signal(not_full)
    ✅管程自动保证进出函数互斥，不用额外加mutex互斥信号量。

（三）signal两种实现（考试选择）

  1. Hoare管程（立即唤醒）：signal后，被唤醒进程优先抢占管程，发送signal的进程阻塞；
  2. Mesa管程（常用，延后唤醒）：signal后，发送信号的进程继续占用管程，被唤醒进程回到入口就绪队列排队（主流编程语言Java/C#采用）。

本章高频考点总结✅

  1. 名词解释：临界区、忙等自旋、记录型信号量、死锁、饥饿
  2. 简答：三大临界准则；自旋锁优缺点；死锁四大条件
  3. 编程大题：生产者消费者/读者写者，用wait/signal写伪代码

第七章 死锁Deadlock期末考点梳理

一、死锁四大必要条件（第一张PPT，名词+简答必背📌）

四个条件必须同时全部满足才会发生死锁，破坏任意一条即可破除死锁

  1. Mutual exclusion 互斥条件：临界资源一次只能被一个进程占用，无法共享使用。
  2. Hold and wait 保持并等待：进程已经占有部分资源，同时继续等待其他被别的进程占用的资源。
  3. No preemption 非剥夺：已分配给进程的资源，系统不能强行收回，只能进程主动释放。
  4. Circular wait 环路等待：进程-资源形成环形依赖：$P_1$等$P_2$资源、$P_2$等$P_3$……$P_n$等$P_1$资源。

破环死锁思路：死锁预防=静态破坏四大条件之一；死锁避免=动态预判拒绝不安全分配；死锁检测+解除=出现死锁后处理。

二、资源分配图Resource-Allocation Graph（第二张，识图选择题💡）

图形规则

  • 方框：资源类型，方框内圆点=该资源实例个数；圆圈=进程
  • 资源→进程：资源已分配给该进程；进程→资源：进程申请该资源

判死锁结论

  1. 图无任何环路 → 一定无死锁
  2. 图存在环路：
    • 环路里每种资源只有1个实例：环路=死锁，环内进程全部死锁
    • 环路里资源存在多个实例：不一定死锁（剩余资源可满足某个进程申请、打破环路）

三、两种死锁避免算法（第三张，计算题核心⚙️）

1. 资源分配图避免算法

  • 适用限制：每种资源只能有单个实例，多实例资源无法使用；分配资源后若产生环路则拒绝本次分配。

2. 银行家算法Banker’s Algorithm（考试重中之重，大题必考）

适用场景：各类资源可拥有任意数量实例

核心概念：安全状态Safe State

存在安全序列：按照该序列依次给每个进程分配所需全部资源，所有进程都能顺利完成、释放资源，系统不会进入死锁；分配资源后系统不安全则拒绝本次资源申请。

四大数组（做题固定）

  1. Available[]：系统当前剩余各类资源总数
  2. Max[][]：每个进程总共需要各类资源最大数量
  3. Allocation[][]：每个进程已经分到的资源数量
  4. Need[][] = Max - Allocation：进程还缺的资源数

算法步骤

  1. 收到进程$P_i$资源请求Request；
  2. 检查Request ≤ Need：不满足则申请非法；
  3. 试探分配：Available=Available-Request，Allocation+=Request，Need-=Request；
  4. 寻找安全序列：遍历进程，找到满足Need ≤ Available的进程，假设其完成并归还资源（Available+=Allocation）；
  5. 能找到完整安全序列→分配生效；找不到→撤销试探，拒绝分配。

本章考点总结✅

  1. 名词：死锁、安全状态、银行家算法、环路等待
  2. 简答：死锁四大必要条件、死锁四种处理方式（预防/避免/检测/解除）
  3. 选择：资源分配图判死锁、四大条件辨析
  4. 大题：银行家算法，求Need矩阵+判断安全序列+资源申请是否可分配

第八章 主存管理期末考点梳理

一、基础概念（第一张PPT📌）

1. 逻辑地址 & 物理地址

  • 逻辑地址（虚地址）：CPU编译/运行产生、相对于进程起始0编址，用户编程可见；
  • 物理地址（实地址）：内存真实硬件地址，由MMU地址转换生成，程序不可直接操控。

2. 地址绑定Address binding（三种时机）

  1. 编译时绑定：编译直接生成物理地址，程序必须固定内存位置，不可换址；
  2. 装入时绑定：装入内存时确定物理地址，进程在内存中不能移动；
  3. 运行时绑定（动态重定位，现代OS）：运行期间动态转换地址，进程可换入换出（对应交换Swapping）。

3. 交换Swapping（中程调度）

内存紧张时，把就绪/阻塞进程整体换出到磁盘（挂起），腾出内存；内存空闲再换回内存运行，缓解内存不足。

二、基址-限长寄存器寻址（第二张图💡）

  • Relocation register(基址/重定位寄存器)：存放进程在内存的起始物理地址；
  • Limit register(限长寄存器)：存放进程逻辑地址最大值（进程大小）；
  1. 合法性检查：逻辑地址 < 限长值，不满足→越界陷阱报错；
  2. 地址换算：物理地址 = 基址 + 逻辑地址；
    ✅实现进程地址隔离、内存保护，是连续分配的硬件基础。

三、连续内存分配（3、4页⚙️）

所有进程占用一段连续整块内存，分两类：

1. 固定分区Fixed partition

内存提前划若干固定大小分区，一个分区装一个进程；
❌产生内部碎片（分区剩余空间无法分给别的进程），无外部碎片。

2. 可变分区Dynamic partition（分区大小随进程按需分配）

三大分配算法（计算题必考）

  1. 首次适应First-fit：从内存低地址开始找第一个够大空闲块，常用；
  2. 最佳适应Best-fit：找刚好够用的最小空闲块，碎片极小块多；
  3. 最坏适应Worst-fit：找最大空闲块拆分，减少细碎小碎片。

两类碎片

  • 内部碎片：分配给进程的空间 > 进程实际需要，块内空余；固定分区典型；
  • 外部碎片：空闲内存分散成大量不连续小块，总和够但单个不够分配；可变分区典型，紧凑(Compaction)可整理消除外碎片。

四、分页存储Paging（重点大题，5~7页）

基础名词

  • 页Page：进程逻辑空间等长拆分，逻辑页；
  • 页框Frame：物理内存同等大小拆分，物理块；

一页刚好装入一个页框，分页彻底消除内部碎片以外的外碎片，仅最后一页存在少量内碎片。

普通分页地址转换

逻辑地址 = 【页号p | 页内偏移d】

  1. 用p查页表→对应页框号f；
  2. 物理地址 = f×页长 + d；

缺点：访存需要两次内存访问：①查页表 ②读写数据，速度慢。

TLB快表（联想寄存器，解决两次访存开销）

TLB高速缓存常用「页号→页框号」映射：

  1. TLB命中(hit)：快表直接拿到页框，1次访存完成；
  2. TLB缺失(miss)：去内存查页表，更新TLB，共2次访存。

有效访问时间EAT计算公式（最后一页计算题）

$$EAT=hit_rate×t_1+(1-hit_rate)×t_2$$
例题：TLB命中0.8，命中耗时120ns、缺失220ns
$EAT=0.8×120+0.2×220=\boldsymbol{140\ ns}$

本章考点总结✅

  1. 名词：逻辑地址、动态重定位、内外碎片、TLB、分页
  2. 简答：连续与分页优缺点、三种分区分配算法对比
  3. 计算：地址转换、EAT有效访问时间、分区分配内存计算

第九章 虚拟内存Virtual Memory考点梳理💡

一、虚拟内存基础（第1页）

核心定义

虚拟内存依靠请求分页，让逻辑地址空间 >> 物理内存大小；程序无需全部装入内存，仅把当前运行需要的页面调入内存，剩余页面存放在磁盘对换区，从逻辑上扩充内存容量。

二、请求分页Demand Paging（第2、3页，缺页中断流程必考）

1. 请求分页规则

用到哪一页才把该页从磁盘调入内存，不提前装入整道程序；页表增加有效位(存在位)：1=页面在内存，0=页面在外存。

2. 缺页中断Page Fault

访问有效位=0的页面 → 触发缺页陷阱，进入缺页中断服务，中断处理完成后指令重新执行（指令复执Restart instruction）。

3. 缺页完整处理流程（流程图）

  1. 逻辑地址拆分页号+页内偏移，越界则报错；
  2. 查页表，页面在内存：修改访问位、直接访存；
  3. 页面不在内存→缺页中断：
    • 内存有空页框：从磁盘调入页面、更新页表；
    • 内存无空闲框：调用页面置换算法淘汰一页；
      • 淘汰页修改过（修改位=1）：先写回磁盘，再释放；
      • 淘汰页未修改：直接释放页框；
    • 新页装入空框、改写页表，返回原指令重新访问。

三、五大页面置换算法（大题计算题核心，第4页）

已知页面访问序列、内存物理块数，统计缺页次数、缺页率。

  1. OPT最优置换：淘汰未来最长时间不再访问的页面，缺页率最低；理论算法、无法实现，用作基准对比。
  2. FIFO先进先出：按页面装入内存先后，淘汰最早进入的页面；Belady异常：物理块变多，缺页次数反而上升。
  3. LRU最近最少使用：淘汰最久没被访问的页面；无Belady异常，现代OS主流。
    • 近似：Clock时钟(二次机会)：利用访问位，指针循环扫描，访问位1→置0、跳过；访问位0→淘汰；改进时钟：结合修改位优先淘汰干净页。
  4. LFU最少使用：淘汰访问计数最小页面；冷门页长期驻留、热门页频繁换入。
  5. MFU最多使用：淘汰访问次数最多页面，规避LFU冷门常驻缺陷。

四、抖动Thrashing（第5页）

1. 抖动定义

系统频繁缺页、大量CPU耗费在页面换入换出，CPU利用率急剧下降，几乎无法推进有效进程。

成因

进程分配物理页框过少，频繁置换页面；全局置换易引发抖动，局部置换限制单进程页面、抑制抖动。

两种解决策略

  1. 工作集Working-set：跟踪进程一段时间访问页面集合，按工作集大小分配页框；
  2. 缺页频率PFF：设置缺页率阈值，缺页过高→增配页框，缺页过低→回收多余页框。

本章考试考点总结✅

  1. 名词：虚拟内存、缺页中断、LRU、抖动、工作集
  2. 简答：缺页中断处理步骤、FIFO Belady异常、抖动成因与解决
  3. 计算：给定页面走向+内存块数，分别算FIFO/LRU/OPT缺页次数、缺页率

第十章 文件系统接口 考点梳理📌

一、文件基础与三种存取方式

1. 顺序访问 Sequential Access

文件记录按先后次序连续读写，只能从头部向后顺序遍历；磁带是典型顺序存储介质，随机访问效率极差。

2. 直接访问（随机存取）Direct Access

按逻辑块号任意跳转读写指定块，无需遍历前面内容；磁盘默认采用直接访问，数据库、大文件常用。

3. 索引访问 Index Access

单独建立索引表：索引项保存「关键字→数据块地址」，先查索引再定位数据；适合按关键词检索（如字典、数据库索引文件）。

二、五类目录结构（考试简答、选择重点）

1. 单级目录 Single-Level

全系统只有一张目录表，所有文件并列；
✅实现简单；❌全系统文件名不能重名，多用户环境冲突严重。

2. 两级目录 Two-Level

分为：用户主目录+用户文件目录，不同用户目录隔离；
同文件名在不同用户下可共存，解决跨用户重名，但同一用户内部不能重名。

3. 树形目录 Tree-Structured（Linux/Windows主流）

多级分层目录，唯一路径名，无共享硬链接；
✅分级管理、分类清晰；❌同一文件无法被多个目录共享。

4. 无环图目录 Acyclic-Graph

在树形基础上增加硬链接，多个目录可指向同一个实体文件，无环路；
实现文件共享，删除文件时需计数引用，引用计数为0才真正删除。

5. 通用图目录 General-Graph

允许目录互相链接，出现环路；
❌存在循环引用、删除回收困难，易产生磁盘碎片与垃圾文件。

本章考点小结✅

  1. 名词：顺序存取、随机存取、索引存取、硬链接
  2. 简答：五种目录结构优缺点对比、适用场景
  3. 选择：区分各类目录能否重名、是否支持文件共享

第十一章 文件系统实现 考点梳理📌

一、文件系统五层分层结构（从上→下）

应用程序 → 逻辑文件系统 → 文件组织模块 → 基本文件系统 → I/O控制层 → 硬件设备

  1. 逻辑文件系统：管理元数据（FCB、目录、权限），处理文件名、文件属性；
  2. 文件组织模块：逻辑块↔物理块映射，负责空间分配；
  3. 基本文件系统：向设备驱动下发读写块指令；
  4. I/O控制层：设备驱动、中断处理，对接磁盘硬件。

二、三种磁盘文件物理分配方式（必考简答/计算）

1. 连续分配 Contiguous allocation

文件占用磁盘一组连续物理块，目录存起始块号+文件长度。
✅优点：随机访问极快、寻道少；
❌缺点：产生外部碎片，文件扩容困难。

2. 链接分配 Linked allocation

每个磁盘块存「数据+下一块指针」，目录只存首块地址；分隐式链接、显式链接(FAT文件分配表)。
✅无外碎片、易扩容；
❌随机访问低效，要顺着指针遍历；FAT存内存提升查找速度。

3. 索引分配 Indexed allocation

单独设索引块：索引块存放全部数据块物理地址，目录存索引块地址；细分：单级索引、多级索引、混合索引（UNIX inode）。
✅随机访问高效、无外碎片；
❌索引块占用额外磁盘空间。

三、三种空闲磁盘空间管理方法

1. 位示图 Bit vector（位图）

磁盘每一块用1bit标记：0=空闲、1=已占用；
优点：空间开销小、查找分配快；计算题常考：由块号换算位示图下标。

2. 空闲链表 Free list

所有空闲块用指针连成单向链表，目录保存首空闲块号；
✅实现简单；❌遍历分配效率低。

3. 成组链接 Grouping（UNIX超级块算法）

把空闲块按组划分：每组若干空闲块，首块保存下一组的块号集合；兼顾链表与位图优势，系统主流。

本章考点总结✅

  1. 名词：FAT、索引分配、位示图、成组链接
  2. 简答：连续/链接/索引三种分配优缺点对比
  3. 计算：位示图块号计算、混合索引寻址计算

第十二章 大容量存储系统考点梳理📌

一、磁盘调度算法（计算题核心，六类算法，求总寻道长度、平均寻道长度）

寻道：磁头移动到目标磁道，寻道时间是磁盘IO主要耗时

1. FCFS 先来先服务

按IO请求到达先后顺序访问磁道；
✅公平简单；❌磁道随机跳跃，平均寻道距离大。

2. SSTF 最短寻道优先

优先选取离当前磁头位置磁道差最小的请求；
✅平均寻道短；❌饥饿问题，边远磁道长期得不到访问。

3. SCAN（电梯算法）

磁头沿一个方向走到磁盘最边缘，返程反向扫描；
类似电梯上下运行，消除饥饿；缺点：两端磁道等待偏长。

4. C-SCAN 循环扫描

单向走到尽头后，直接跳转到磁盘最内侧起点，继续同方向扫描；
所有磁道等待时间均匀，消除两端等待不均问题。

5. LOOK（改良SCAN）

不用走到磁盘物理边界，走到当前最远端请求就掉头，减少无效空走。

6. C-LOOK（改良C-SCAN）

走到当前最远请求即折返跳转，无多余空行程，工业系统最常用。

二、各级RAID阵列（选择/简答，P=校验盘）

  1. RAID0 条带化：无冗余校验，多盘拆分数据；速度最快，一块盘损坏全阵列数据丢失。
  2. RAID1 镜像：两块盘互为完整备份；100%冗余，可坏一块，成本翻倍。
  3. RAID2：位级海明校验，现已淘汰。
  4. RAID3：比特交错奇偶校验，单块校验盘；按位拆分，面向流式数据。
  5. RAID4：块级集中校验，固定单独一块盘存P；校验盘成为性能瓶颈。
  6. RAID5 分布式奇偶校验：校验块分散在所有磁盘；最多坏1块盘，性价比最高，企业主流。
  7. RAID6 双校验(P+Q)：两份独立校验信息，可同时损坏两块磁盘，容错更强。

本章考点总结✅

  1. 名词：寻道时间、电梯算法、RAID5、条带化
  2. 计算题：给定磁道序列+磁头起始位置，分别算6种调度总寻道长度
  3. 简答：各RAID优缺点、容错能力对比

第十三章 I/O系统三大I/O控制方式考点📌

1. 轮询 Polling（程序直接控制I/O）

CPU循环不断查询设备状态寄存器，设备未就绪则CPU空等循环。

  • 优点：逻辑简单、硬件成本低；
  • 缺点：CPU全程忙等，资源浪费严重，只适合低速外设。

2. 中断驱动 Interrupt

设备就绪后主动向CPU发中断信号，CPU暂停当前进程，转入中断服务程序完成数据传输。

  • 流程：CPU发起I/O后继续执行原有任务 → 设备完成触发中断 → CPU现场保护→I/O读写→恢复现场；
  • 优点：CPU无需空轮询，利用率大幅提升；
  • 局限：每次只传输一个字/字节，大批量数据频繁触发多次中断，开销大。

3. DMA 直接存储器访问

DMA控制器接管总线，整块数据在内存与外设之间直接传输，全程不需要CPU逐字搬运；仅在整批数据传送完毕后触发一次中断通知CPU。

  • 工作：CPU预置DMA参数（源地址、目的地址、传输长度）后返回作业，DMA独立完成批量数据搬运；
  • 优点：大批量高速I/O最优，中断次数极少，CPU开销最小；磁盘等块设备标配。

三种方式对比小结

  1. 轮询：CPU全程忙等，低速小设备；
  2. 中断：单字节传输、多次中断，字符设备（键盘鼠标）；
  3. DMA：整块DMA自动搬运、一次中断，高速块设备（磁盘）。
